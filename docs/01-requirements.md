# 需求分析

## 背景

实验室有多台共享服务器，支持多用户登录。成员在使用前通常需要判断哪台机器空闲、GPU 是否被占满、某个用户是否占用了大量资源。如果只靠口头询问或手动登录服务器执行命令，效率低，也容易产生资源冲突。

因此需要一个网页端小工具，把局域网服务器的当前资源状态集中展示出来。

## 核心目标

1. 展示服务器在线/离线状态。
2. 展示每台服务器在线用户数、总用户数。
3. 展示 CPU、内存、GPU 总体占用情况。
4. 展示哪个用户的哪个进程占用了多少 CPU、内存、GPU、显存。
5. 支持在网页端手动添加局域网服务器。
6. 支持 Linux 和 Windows 混合环境。
7. 采集端保持轻量，部署步骤尽量简单。

## 功能需求

### 1. 服务器管理

- 新增服务器：名称、IP/域名、系统类型、位置、用途、GPU 类型、备注。
- 查看服务器列表：在线状态、最近心跳时间、Agent 版本、采集状态。
- 编辑服务器信息。
- 禁用或删除服务器。
- 生成 Agent 注册 Token 和安装参数。

### 2. 总览看板

- 显示服务器总数、在线服务器数、在线用户总数。
- 显示每台服务器 CPU/内存/GPU 摘要。
- 用颜色区分空闲、较忙、满载、离线。
- 支持自动刷新，MVP 推荐 5-10 秒刷新一次。

### 3. 用户与会话

- 展示每台服务器当前登录用户。
- 展示每个用户的登录来源、登录时间、会话数量。
- Linux 可从 `who`、`w` 等命令采集。
- Windows 可从 `quser`、WMI/CIM 等方式采集。

### 4. 进程资源占用

- 展示进程 PID、用户名、命令/进程名、CPU 占用、内存占用、启动时间。
- 支持按 CPU、内存、GPU 显存排序。
- 支持按服务器、用户、关键字筛选。
- 命令行参数默认可见，因为需求是实验室成员都能查看资源具体情况；后续如有隐私要求，可增加命令参数脱敏开关。

### 5. GPU 监控

- 展示每张 GPU 的编号、名称、温度、功耗、GPU 利用率、显存总量、显存已用。
- 展示 GPU 上运行的进程、PID、用户名、显存占用。
- Linux A100 和 Windows P100 优先支持 NVIDIA GPU，通过 `nvidia-smi` 或 NVML 采集。
- 普通显卡主机可只展示基础 GPU 信息，采集不到时不影响整体页面。

### 6. Agent 状态

- Agent 首次启动后向后端注册。
- Agent 定期心跳，后端据此判断服务器在线状态。
- Agent 定期上报资源快照。
- Agent 异常、版本过低、采集失败时在页面提示。

## 非功能需求

| 类型 | 要求 |
|---|---|
| 易部署 | Agent 配置项少，尽量一条命令启动或注册为系统服务 |
| 低侵入 | 默认只读采集，不修改服务器任务，不杀进程 |
| 跨平台 | 支持 Linux 和 Windows |
| 实时性 | 当前状态 5-10 秒级刷新即可，不追求毫秒级 |
| 可维护 | 后端统一数据模型，OS 差异封装在 Agent 内 |
| 安全性 | Agent 使用 Token 注册，后端不保存服务器管理员密码 |
| 可扩展 | 后续可加告警、历史趋势、资源统计、用户登录体系 |

## MVP 范围

MVP 必须完成：

- 手动添加服务器。
- Agent 注册、心跳、快照上报。
- Linux A100 服务器完整采集。
- 至少一台 Windows 服务器基础采集。
- 服务器列表、总览看板、进程列表、GPU 详情页。

MVP 暂不做：

- 远程杀进程。
- 作业调度。
- 配额计费。
- 复杂告警规则。
- 长期高精度历史曲线。
- 与 LDAP/校园统一身份认证集成。

## 风险点

1. Windows 上获取进程所属用户可能需要管理员权限。
2. GPU 进程 PID 与系统进程 PID 需要做映射，进程退出太快时可能出现短暂缺失。
3. `nvidia-smi` 输出在不同驱动版本上字段可能略有差异，需要 Agent 做兼容。
4. 如果采集频率太高，会给服务器和数据库增加不必要负担。
5. 全员可见进程命令行可能涉及隐私，建议先在实验室内部达成共识。
