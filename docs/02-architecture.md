# 总体架构设计

## 方案对比

| 方案 | 说明 | 优点 | 缺点 | 结论 |
|---|---|---|---|---|
| 后端远程轮询 | Spring Boot 通过 SSH/WinRM 连接服务器采集 | 不需要部署 Agent | 需要保存凭据，Windows 配置复杂，权限和稳定性较差 | 仅作为后备方案 |
| 轻量 Agent 上报 | 每台服务器部署本机 Agent，主动上报后端 | 跨平台清晰，安全边界好，采集准确 | 每台服务器需要安装一次 Agent | 推荐方案 |
| 混合模式 | 优先 Agent，特殊机器用远程轮询 | 灵活 | 架构复杂度较高 | 后续增强 |

## 推荐架构

```text
┌────────────────────┐
│ Vue Web 前端        │
│ 看板 / 服务器 / 进程 │
└─────────┬──────────┘
          │ REST API
┌─────────▼──────────┐
│ Spring Boot 后端    │
│ 认证 / 服务器管理    │
│ 快照接收 / 查询 API  │
└──────┬────────┬─────┘
       │        │
       │        └────────── 数据库 MySQL 5.7.35
       │
       │ Agent API
┌──────▼──────────────┐
│ Linux Agent          │
│ A100 服务器           │
└──────────────────────┘
┌──────────────────────┐
│ Windows Agent         │
│ CPU 主机 / P100 主机   │
└──────────────────────┘
```

## 模块划分

### 1. 前端 `web`

技术建议：

- Vue 3
- Vite
- TypeScript
- Pinia
- Vue Router
- ECharts
- Element Plus 或 Naive UI

主要页面：

- 登录页。
- 总览看板。
- 服务器列表。
- 服务器详情。
- 进程列表。
- GPU 详情。
- Agent 安装向导。

### 2. 后端 `server`

技术建议：

- Java 17
- Spring Boot 3.x
- Spring Security
- MyBatis-Plus 或 Spring Data JPA
- MySQL 5.7.35 作为 MVP 默认数据库
- Redis 可选，用于缓存最新快照和在线状态

核心职责：

- 用户登录与权限。
- 服务器资产管理。
- Agent 注册 Token 管理。
- 接收 Agent 心跳和资源快照。
- 维护最新状态和短期历史。
- 提供前端查询 API。

### 3. 采集端 `agent`

技术建议：

- 首选 Java 轻量 Agent，与后端共享 DTO 和签名逻辑。
- Agent 不需要嵌入 Web 服务，只需要定时采集并 HTTP 上报。
- Linux 注册为 `systemd` 服务。
- Windows 先支持命令行运行，后续用 WinSW 或 NSSM 注册为 Windows Service。

核心职责：

- 获取本机 hostname、系统类型、CPU、内存、GPU 基础信息。
- 获取在线用户和进程列表。
- 获取 NVIDIA GPU 与 GPU 进程信息。
- 将采集结果转换为统一快照格式。
- 周期性上报后端。

## 数据流

### 服务器添加与 Agent 注册

1. 管理员在网页端新增服务器。
2. 后端创建服务器记录，生成一次性注册 Token。
3. 页面展示 Agent 安装命令。
4. 管理员在目标服务器执行命令并启动 Agent。
5. Agent 使用注册 Token 调用后端注册接口。
6. 后端返回长期 Agent 凭证。
7. Agent 后续使用长期凭证心跳和上报。

### 状态采集与展示

1. Agent 每 5-10 秒采集一次本机状态。
2. Agent 上报快照到后端。
3. 后端保存最新快照，必要时写入历史表。
4. Vue 前端定时拉取总览和详情接口。
5. 页面展示服务器、用户、进程、GPU 状态。

## 权限模型

MVP 使用两类角色：

| 角色 | 权限 |
|---|---|
| 普通成员 | 查看所有服务器状态、用户、进程、GPU 信息 |
| 管理员 | 普通成员权限 + 新增/编辑/删除服务器 + 生成 Agent Token |

## 在线状态判断

- Agent 每隔 5 秒发送心跳。
- 后端记录 `last_heartbeat_at`。
- 超过 20 秒无心跳：标记为 `WARN`。
- 超过 60 秒无心跳：标记为 `OFFLINE`。

具体阈值可在配置文件中调整。

## 历史数据策略

MVP 不做长期高精度时序数据库，只保留：

- 最新快照：用于看板实时展示。
- 短期历史：保留 7 天，按 10-30 秒粒度存储，用于简单趋势图。

后续如果要做月度统计或高精度趋势，再考虑 TimescaleDB、Prometheus 或 VictoriaMetrics。
